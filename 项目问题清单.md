### 问题 #1：Java 版本不兼容（环境问题）

**严重程度**: 🔴 Critical（P0 - 最高优先级）  
**问题状态**: ⚠️ 未修复  
**影响**: **完全无法编译**，是最根本的阻塞性问题

#### 问题详情

| 项目 | 配置值 | 当前系统值 | 状态 |
|------|--------|------------|------|
| 项目要求的 Java 版本 | 11 (LTS) | - | ⚠️ 未满足 |
| 系统当前 Java 版本 | - | **25.0.2** (2026) | ❌ 过高 |
| AGP 8.10.1 支持的最高版本 | JDK 21 | - | ❌ 超出范围 |
| Gradle 版本 | 8.11.1 | 8.11.1 | ✓ 匹配 |

#### 问题证据

**文件**: `app/build.gradle.kts:52-59`
```kotlin
compileOptions {
    sourceCompatibility = JavaVersion.VERSION_11
    targetCompatibility = JavaVersion.VERSION_11
}
kotlinOptions {
    jvmTarget = "11"
}
```

**错误日志** (`compile_errors.txt`):
```
FAILURE: Build failed with an exception.

* What went wrong:
25.0.2
```

#### 根本原因

1. 项目明确配置使用 **Java 11**（`jvmTarget = "11"`）
2. 系统安装的是 **Java 25.0.2**（2026年最新LTS版本）
3. **AGP 8.10.1 只支持到 JDK 21**，不支持 Java 25
4. Gradle wrapper 调用系统 `java` 命令时，AGP 检查版本发现不匹配，抛出异常

#### 解决方案

**方案一：降级 Java 版本（强烈推荐）**

1. **下载安装 JDK 11 或 JDK 17**（LTS 版本）
   - 推荐下载地址: https://adoptium.net/ (Eclipse Adoptium Temurin)
   - 选择版本: **JDK 17**（推荐）或 **JDK 11**
   - 安装路径: `C:\Program Files\Eclipse Adoptium\jdk-17.0.12+7`

2. **配置 `JAVA_HOME` 环境变量**
   ```cmd
   # 以管理员身份运行 CMD
   setx JAVA_HOME "C:\Program Files\Eclipse Adoptium\jdk-17.0.12+7"
   setx PATH "%JAVA_HOME%\bin;%PATH%"
   ```

3. **重启命令行并验证**
   ```cmd
   # 打开新的命令提示符
   java -version
   
   # 期望输出：
   # openjdk version "17.0.12" 2024-07-16 LTS
   # 或
   # openjdk version "11.0.24" 2024-07-16 LTS
   ```

4. **清理并重新构建项目**
   ```cmd
   cd D:\NetGeoCourier2
   gradlew clean
   gradlew assembleDebug
   ```

**方案二：升级 AGP 版本（不推荐，风险高）**

修改 `gradle/libs.versions.toml`:
```toml

```

⚠️ **风险警告**: 
- 可能引发 Compose、Kotlin 等依赖不兼容
- 需要同步升级多个依赖库
- 可能引入新的未知问题

### 问题 #2：高德 SDK 方法名拼写错误

**严重程度**: 🔴 Critical（P0 - 最高优先级）  
**问题状态**: ⚠️ 未修复  
**影响**: 运行时崩溃、定位资源泄漏、监听器无法正确移除

#### 问题详情

**文件**: `app/src/main/java/com/example/netgeocourier/helper/LocationHelper.kt`  
**行号**: 142  
**错误代码**:
```kotlin
locationClient?.unRegisterLocationListener(this)  // ❌ 大写 R
```

**正确写法**:
```kotlin
locationClient?.unregisterLocationListener(this)  // ✅ 全小写
```

#### 问题证据

**LocationHelper.kt:139-147** (当前错误代码):
```kotlin
fun stopLocation() {
    try {
        locationClient?.stopLocation()
        locationClient?.unRegisterLocationListener(this)  // ❌ 拼写错误
        locationClient = null
    } catch (e: Exception) {
        e.printStackTrace()
    }
}
```

**高德官方文档确认**:
- 高德定位 SDK 的方法名为 `unregisterLocationListener()`（全小写）
- Java 方法命名规范：动词+名词，首字母小写，驼峰命名
- "unregister" 是一个单词，全部小写

#### 影响分析

1. **编译阶段**: 
   - 可能通过编译（因为 Kotlin/Java 的动态绑定）
   - 但 IDE 可能显示警告

2. **运行阶段**:
   - 当 `stopLocation()` 被调用时
   - 会抛出 `NoSuchMethodError` 异常
   - 导致应用崩溃
   - **定位功能完全失效**

3. **资源泄漏**:
   - 监听器未被正确移除
   - 定位客户端仍在后台运行
   - 电池消耗激增
   - 后续定位请求可能冲突

#### 触发场景

- 定位成功后（`onLocationChanged` 回调中调用 `stopLocation()`）
- 定位超时或失败时
- Activity/Fragment 销毁时（如果调用了 `stopLocation()`）

#### 修复步骤

1. 打开文件: `app/src/main/java/com/example/netgeocourier/helper/LocationHelper.kt`

2. 定位到第 142 行

3. 修改代码:
```diff
- locationClient?.unRegisterLocationListener(this)
+ locationClient?.unregisterLocationListener(this)
```

4. 保存文件并重新编译

**修复后代码**:
```kotlin
fun stopLocation() {
    try {
        locationClient?.stopLocation()
        locationClient?.unregisterLocationListener(this)  // ✅ 已修复
        locationClient = null
    } catch (e: Exception) {
        e.printStackTrace()
    }
}
```

### 问题 #3：双重坐标转换（逻辑错误）

**严重程度**: 🔴 Critical（P0 - 最高优先级）  
**问题状态**: ⚠️ 未修复  
**影响**: 坐标偏差可达 **数百米**，导致地图显示位置严重错误

#### 问题概述

高德定位 SDK 返回的坐标**已经是 GCJ-02（火星坐标系）**，但代码错误地对其进行了第二次 WGS-84 → GCJ-02 转换，导致坐标严重偏离真实位置。

#### 受影响文件

| 文件 | 行号 | 问题类型 |
|------|------|----------|
| `FileHelper.kt` | 34-35 | 保存 CSV 时错误转换 |
| `NetTestScreen.kt` | 411-414 | UI 显示时错误转换 |
| `NetTestScreen.kt` | 475-478 | 历史列表显示时错误转换 |

#### 问题证据

**FileHelper.kt:34-35** (错误代码):
```kotlin
val gcjLat = it.latitude
val gcjLon = it.longitude
out.write("${it.timestamp},$gcjLon,${"%.6f".format(gcjLat)},...")  // ❌ 已转换过
```

**NetTestScreen.kt:411-414** (错误代码):
```kotlin
val (gcjLat, gcjLon) = remember(result) {
    result.latitude to result.longitude  // ❌ 注释说"已经是 GCJ-02"，但实际用了转换函数
}
```

**NetTestScreen.kt:475-478** (错误代码):
```kotlin
val (gcjLat, gcjLon) = remember(result) {
    result.latitude to result.longitude  // ❌ 同上
}
```

#### 数据流转分析

```
1. LocationHelper.getCurrentLocation() 获取 AMapLocation
   ↓
2. AMapLocation.latitude/longitude → 已经是 GCJ-02 坐标 ✅
   ↓
3. 存储到 NetTestResult (lines 106-107 in NetTestScreen or 73-74 in ViewModel)
   ↓
4. FileHelper.saveCsv() 调用 CoordTransform.wgs84ToGcj02() ❌
   - 输入：已经是 GCJ-02 的坐标
   - 输出：错误的坐标（双重转换）
   ↓
5. CSV 文件中的坐标偏差数百米
```

#### 坐标系统说明

| 坐标系 | 说明 | 使用场景 |
|--------|------|----------|
| **WGS-84** | GPS 原始坐标（国际标准） | Google Maps, 国际GPS设备 |
| **GCJ-02** | 火星坐标系（中国标准） | 高德地图, 腾讯地图, 百度地图（部分） |
| **BD-09** | 百度坐标系 | 百度地图 |

**关键事实**:
- ✅ **高德定位 SDK 直接返回 GCJ-02 坐标**
- ❌ **不需要**调用 `CoordTransform.wgs84ToGcj02()` 转换
- ❌ 再次转换会导致坐标**严重偏移**

#### 修复方案

**原则**: 高德返回的坐标已经是 GCJ-02，**直接使用**，无需转换。

##### 修复 #1：FileHelper.kt

**文件**: `app/src/main/java/com/example/netgeocourier/helper/FileHelper.kt`  
**行号**: 34-35

```diff
- val (gcjLat, gcjLon) = CoordTransform.wgs84ToGcj02(it.latitude, it.longitude)
+ val gcjLat = it.latitude      // 高德直接返回 GCJ-02
+ val gcjLon = it.longitude    // 无需再次转换
```

**完整修复代码**:
```kotlin
fun saveCsv(context: Context, list: List<NetTestResult>): String? {
    val dir = getDocumentsDir(context) ?: return null
    val file = File(dir, "nettest.csv")
    val isNewFile = !file.exists()

    FileOutputStream(file, true).bufferedWriter().use { out ->
        if (isNewFile) {
            out.write("timestamp,longitude_gcj02,latitude_gcj02,upload_mbps,download_mbps,ping_ms\n")
        }
        list.forEach {
            // ✅ 直接使用高德返回的 GCJ-02 坐标
            val gcjLat = it.latitude
            val gcjLon = it.longitude
            out.write(
                "${it.timestamp},$gcjLon,${"%.6f".format(gcjLat)}," +
                    "${"%.2f".format(it.upload)},${"%.2f".format(it.download)},${it.ping}\n"
            )
        }
    }
    // ...
}
```

##### 修复 #3：NetTestScreen.kt (位置 #2)

## 🟡 高优先级问题（功能错误）

### 问题 #4：重复初始化防护缺失

**严重程度**: 🟡 High（P1 - 高优先级）  
**问题状态**: ⚠️ 未修复  
**影响**: 定位客户端冲突、资源泄漏、电池消耗激增

#### 问题详情

**文件**: `app/src/main/java/com/example/netgeocourier/helper/LocationHelper.kt`  
**行号**: 62-86

**当前代码**:
```kotlin
suspend fun getCurrentLocation() = suspendCancellableCoroutine { cont ->
    // ❌ 每次都新建客户端，未检查是否已有实例在运行
    locationClient = AMapLocationClient(context.applicationContext).apply {
        setLocationListener(this@LocationHelper)
        setLocationOption(createLocationOption())
        startLocation()
    }
}
```

#### 问题分析

**场景**: 用户快速连续点击"开始定位"按钮

1. 第一次调用 `getCurrentLocation()`
   - 创建 `AMapLocationClient` 实例 A
   - 开始定位

2. 第二次调用 `getCurrentLocation()`（在第一次完成前）
   - 创建 `AMapLocationClient` 实例 B
   - 实例 A 未被停止，仍在运行
   - **两个定位客户端同时工作** ❌

3. 后果:
   - 资源冲突
   - 电池消耗激增（两个 GPS 模块同时工作）
   - 定位回调混乱（哪个客户端先返回？）
   - 可能抛出异常

**即使在单次调用中**:
- 如果用户取消协程（例如旋转屏幕）
- `invokeOnCancellation` 会调用 `stopLocation()`
- 但下一次调用时仍然会创建新实例，旧实例可能未完全清理

#### 解决方案

在创建新客户端前，**先停止并清理旧的客户端**。

**修复代码**:
```kotlin
suspend fun getCurrentLocation() = suspendCancellableCoroutine { cont ->
    // ✅ 先停止之前的定位客户端（如果有）
    locationClient?.stopLocation()
    locationClient?.unregisterLocationListener(this)
    
    if (!PermissionHelper.hasLocationPermission(context)) {
        cont.resume(null)
        return@suspendCancellableCoroutine
    }

    locationContinuation = cont

    try {
        locationClient = AMapLocationClient(context.applicationContext).apply {
            setLocationListener(this@LocationHelper)
            setLocationOption(createLocationOption())
            startLocation()
        }
    } catch (e: Exception) {
        e.printStackTrace()
        cont.resume(null)
    }

    cont.invokeOnCancellation {
        stopLocation()
        locationContinuation = null
    }
}
```

**关键改进**:
1. 在创建新实例前，先调用 `stopLocation()` 停止旧实例
2. 调用 `unregisterLocationListener()` 移除旧监听器
3. 确保只有一个定位客户端同时运行

## 🟢 中优先级问题（优化建议）

### 问题 #5：错误信息丢失

**严重程度**: 🟢 Medium（P2 - 中优先级）  
**问题状态**: ⚠️ 未修复  
**影响**: 定位失败时无法区分具体原因，用户体验差

#### 问题详情

**文件**: `LocationHelper.kt:124-127`

**当前代码**:
```kotlin
override fun onLocationChanged(location: AMapLocation?) {
    stopLocation()

    location?.let {
        if (it.errorCode == 0) {
            // 定位成功
            locationContinuation?.resume(it)
        } else {
            // 定位失败
            it.errorCode.let { errorCode ->
                android.util.Log.e("LocationHelper", "定位失败, 错误码: $errorCode, 错误信息: ${it.errorInfo}")
            }
            // ❌ 只打印日志，未传递错误信息
            locationContinuation?.resume(null)  // 调用方无法知道具体错误
        }
    } ?: run {
        locationContinuation?.resume(null)  // ❌ 无法区分是失败还是权限问题
    }

    locationContinuation = null
}
```

#### 问题影响

调用方 `NetTestViewModel.doTest()`:
```kotlin
val location = locationHelper.getCurrentLocation()
if (location == null) {
    Toast.makeText(context, "Failed to get location.", Toast.LENGTH_SHORT).show()
    // ❌ 不知道是权限被拒、GPS关闭、还是网络问题
    // 无法给出针对性的错误提示
}
```

#### 改进建议

**方案一：使用 Result 类型（推荐）**

定义密封类表示定位结果：
```kotlin
sealed class LocationResult {
    data class Success(val location: AMapLocation) : LocationResult()
    data class Failure(val errorCode: Int, val errorInfo: String) : LocationResult()
    object PermissionDenied : LocationResult()
    object Unknown : LocationResult()
}
```

修改 `getCurrentLocation()` 返回类型：
```kotlin
suspend fun getCurrentLocation(): LocationResult = suspendCancellableCoroutine { cont ->
    // ...
    override fun onLocationChanged(location: AMapLocation?) {
        stopLocation()
        when {
            location == null -> cont.resume(LocationResult.Unknown)
            it.errorCode == 0 -> cont.resume(LocationResult.Success(location))
            it.errorCode == AMapLocation.LOCATION_FAILURE_PERMISSION_DENIED ->
                cont.resume(LocationResult.PermissionDenied)
            else -> cont.resume(
                LocationResult.Failure(it.errorCode, it.errorInfo ?: "Unknown error")
            )
        }
        locationContinuation = null
    }
}
```

**方案二：抛出自定义异常（简化版）**

```kotlin
class LocationException(val errorCode: Int, message: String) : Exception(message)

override fun onLocationChanged(location: AMapLocation?) {
    stopLocation()
    when {
        location == null -> locationContinuation?.resume(null)
        it.errorCode == 0 -> locationContinuation?.resume(location)
        else -> locationContinuation?.resumeWithException(
            LocationException(it.errorCode, it.errorInfo ?: "Location failed")
        )
    }
    locationContinuation = null
}
```

### 问题 #6：网络错误处理不完善

**严重程度**: 🟢 Medium（P2 - 中优先级）  
**问题状态**: ⚠️ 未修复  
**影响**: 网络请求失败时用户体验不佳

#### 问题详情

**文件**: `EvaluationRepository.kt:11-25`

**当前代码**:
```kotlin
suspend fun getEvaluation(): Result<EvaluationData> = withContext(Dispatchers.IO) {
    runCatching {
        val response = apiService.getEvaluation()
        if (!response.isSuccessful) {
            throw IOException("Request failed: HTTP ${response.code()}")
        }
        // ...
    }
}
```

**问题**:
1. 错误信息过于简单（只有 HTTP 状态码）
2. 没有区分网络超时、连接失败、服务器错误等场景
3. 没有重试机制
4. 用户看到的提示不友好

#### 改进建议

可以扩展错误处理，提供更详细的错误信息：
```kotlin
sealed class NetworkError {
    object Timeout : NetworkError()
    object ConnectionFailed : NetworkError()
    data class HttpError(val code: Int, val message: String) : NetworkError()
    data class ParseError(val cause: Throwable) : NetworkError()
    object NoToken : NetworkError()
}

suspend fun getEvaluation(): Result<EvaluationData> = withContext(Dispatchers.IO) {
    runCatching {
        try {
            val response = apiService.getEvaluation()
            if (response.isSuccessful) {
                val body = response.body()
                if (body?.success == true && body.data != null) {
                    Result.success(body.data)
                } else {
                    Result.failure(IOException(body?.message ?: "Request failed"))
                }
            } else {
                Result.failure(IOException("HTTP ${response.code()}"))
            }
        } catch (e: IOException) {
            Result.failure(NetworkError.ConnectionFailed)
        } catch (e: Exception) {
            Result.failure(NetworkError.ParseError(e))
        }
    }
}
```
